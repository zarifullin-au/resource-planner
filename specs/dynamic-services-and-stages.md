# Спецификация: Динамические виды услуг и этапы в нормативах

figma_component (if used): N/A

## Краткое описание

Сейчас списки видов услуг (`SERVICES`) и этапов (`STAGES`) — это захардкоженные массивы в `lib/calc.ts`. Пользователь не может добавить новый вид услуги или этап без правки исходного кода. Эта фича переносит виды услуг и этапы в базу данных, чтобы ими можно было управлять со страницы «Нормативы», а все расчёты нагрузки автоматически учитывали нормы, введённые для новых услуг и этапов.

## Функциональные требования

### 1. Новые модели данных — таблицы `Service` и `Stage`

Добавить две новые Prisma-модели:

```
model Service {
  id    String @id @default(cuid())
  name  String @unique
  color String @default("#6b7280")
  order Int    @default(0)
}

model Stage {
  id    String @id @default(cuid())
  name  String @unique
  order Int    @default(0)
}
```

При первой миграции заполнить их текущими захардкоженными значениями:
- Виды услуг: `ДПИ`, `ЭАП`, `АЛР`, `Авторский надзор` (с существующими цветами)
- Этапы: `Этап 1`, `Этап 2`, `Этап 3`, `Этап 4`

### 2. API-маршруты

- `GET /api/services` — вернуть все виды услуг, отсортированные по полю `order`
- `POST /api/services` — создать новый вид услуги (обязательное поле `name`, необязательное `color`)
- `GET /api/stages` — вернуть все этапы, отсортированные по полю `order`
- `POST /api/stages` — создать новый этап (обязательное поле `name`)

Эндпоинты удаления и переименования в данном скоупе не предусмотрены — они рискуют сломать каскадные связи с уже существующими нормами и договорами.

### 3. `useAppData` — загрузка видов услуг и этапов

Добавить `services` и `stages` к пяти параллельным запросам в `lib/useAppData.ts` и экспортировать наряду с остальными данными.

### 4. `lib/calc.ts` — замена захардкоженных массивов

- Удалить экспорты `SERVICES` и `STAGES`.
- Оставить их как резервные константы (`DEFAULT_SERVICES`, `DEFAULT_STAGES`) — только для сида/миграции и тестов.
- `calcStageHours()` и `calcLoad()` уже сопоставляют нормы по строковому значению `stage` и `service`, поэтому логику расчётов менять не нужно — она работает с любыми строками.
- Все места, которые сейчас импортируют `SERVICES`/`STAGES` из `lib/calc.ts`, должны быть переведены на динамические списки из `useAppData`.

### 5. UI страницы нормативов — кнопки «+ Вид услуги» и «+ Этап»

Добавить две новые кнопки в правый слот `PageHeader` (рядом с существующей кнопкой «+ Добавить задачу»):

- **«+ Вид услуги»** — открывает небольшой модал с текстовым полем `name` и выбором цвета (color-picker или фиксированная палитра). При сохранении: `POST /api/services`, затем `refresh()`. Новый вид услуги сразу появляется в фильтрах и в выпадающем списке «Вид услуги» в модале добавления нормы.
- **«+ Этап»** — открывает небольшой модал только с полем `name`. При сохранении: `POST /api/stages`, затем `refresh()`. Новый этап сразу появляется в фильтрах и в выпадающем списке «Этап».

Оба модала переиспользуют существующий компонент `Modal` из `components/ui/index.tsx`.

### 6. Модал добавления нормы — динамические списки

Элементы `<select>` для «Вид услуги» и «Этап» в существующем модале добавления нормы должны использовать динамические списки `services` и `stages` из `useAppData` вместо захардкоженных массивов `SERVICES`/`STAGES`.

### 7. Модал создания / редактирования договора — динамические списки

Выпадающие списки вида услуги и этапа в модале договоров (`components/contracts/`) тоже должны быть переведены на динамические списки из `useAppData`.

### 8. Кнопки фильтрации — динамические списки

Массивы `svcOptions` и `stageOptions` на странице нормативов сейчас строятся из `SERVICES`/`STAGES`. Их нужно перестроить на основе динамических списков.

### 9. Пересчёт нагрузки

Дополнительных изменений не требуется: `calcLoad()` — чистая функция, которая ищет нормы по строкам `service`+`stage`. Как только новые нормы для нового вида услуги или этапа сохранены, любой договор, ссылающийся на эти строки, автоматически учтёт их в расчётах нагрузки после следующего `refresh()`.

## Возможные граничные случаи

- **Дублирующееся название**: `POST /api/services` и `POST /api/stages` должны возвращать 409, если имя уже существует (ограничение `@unique` это обеспечит на уровне БД; нужно показать понятную ошибку через `showError`).
- **Пустое название**: перед отправкой проверять, что `name.trim()` не пустой.
- **Договоры, ссылающиеся на услугу/этап, которых нет в новых таблицах**: существующие договоры хранят `service` и `stage` как обычные строки. После миграции нужно одноразово сделать upsert, чтобы все строковые значения, уже встречающиеся в таблицах `Contract` и `ContractStage`, были представлены в новых таблицах `Service`/`Stage`.
- **`SERVICE_COLORS` на странице нормативов**: сейчас это захардкоженный словарь в `page.tsx`. Нужно заполнять его динамически из поля `color` модели `Service`; для неизвестных услуг — fallback `#6b7280`.
- **SlotFinder и Timeline**: `slotFinder.ts` и компоненты таймлайна импортируют `STAGES`/`SERVICES` из `lib/calc.ts` и передают в чистые функции. Их нужно обновить: принимать динамические списки как параметры или через контекст/хук.

## Критерии приёмки

- [ ] Пользователь нажимает «+ Вид услуги» на странице нормативов, вводит название (и при желании цвет) — новый вид услуги сразу появляется в фильтрах и в выпадающих списках добавления нормы.
- [ ] Пользователь нажимает «+ Этап», вводит название — новый этап сразу появляется в фильтрах и в выпадающих списках.
- [ ] После добавления норм для нового вида услуги/этапа и их сохранения эти часы отражаются в расчётах нагрузки на страницах «Нагрузка» и «Таймлайн» (изменений кода не требуется — только нормы в БД).
- [ ] При вводе дублирующегося названия отображается понятное сообщение об ошибке, а не крэш.
- [ ] Существующие виды услуг и этапы (засеянные из захардкоженных списков) сохраняются после миграции.
- [ ] Модал создания договора показывает новый вид услуги/этап в своих выпадающих списках.
- [ ] Все кнопки фильтрации на странице нормативов отражают полный динамический список.

## Открытые вопросы

- Нужна ли поддержка переименования или удаления видов услуг? Удаление рискованно, так как на строку имени ссылаются существующие нормы и договоры. Рекомендация: исключить из текущего скоупа; только добавление. Только добавление. 
- Как должна работать сортировка этапов — произвольно пользователем (drag-to-reorder) или в порядке создания? Текущее предложение: порядок создания (авто-инкремент `order`). Перетаскивание — в следующую итерацию. Оставляем drag-to-reorder
- Какой color-picker использовать для новых видов услуг — свободный `<input type="color">` или фиксированная палитра из 6–8 акцентных цветов, соответствующих дизайн-токенам? Фиксированная палитра проще и безопаснее для UI. Фиксированная палитра на выбор. 
- Нужна ли привязка видов услуг к конкретным этапам (например, «Авторский надзор» предполагает только 1 этап, а не 4)? Текущая модель этого не обеспечивает — любой этап может встречаться с любым видом услуги в нормах. Введение такого ограничения вынесено за скоуп. Привязка не нужна.
