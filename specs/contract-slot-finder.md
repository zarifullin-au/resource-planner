# Spec for Contract Slot Finder

figma_component (if used): —

## Summary

Переделать второй блок дашборда (`/timeline`) — сейчас он называется «ОКНО ДЛЯ НОВОГО ДОГОВОРА — СВОБОДНАЯ ЁМКОСТЬ» и фактически дублирует страницы «Загрузка сотрудников» и «Тепловая карта», просто показывая свободные часы по ролям/сотрудникам.

Задача нового блока — отвечать на конкретный вопрос:
**«Когда я могу взять новый договор в работу и с каким составом команды?»**

Пользователь выбирает параметры будущего договора (тип услуги, объект-аналог или его параметры, сложность). Блок подбирает:
1. **Самую раннюю дату старта**, при которой у команды хватит свободных часов на все этапы договора.
2. **Конкретный состав команды** — по одному сотруднику на каждую роль из норм, с учётом их свободной ёмкости и квалификации.
3. Альтернативные варианты состава, если их несколько (разные сотрудники → разные даты старта).

Результат — это «черновик договора», который можно превратить в реальный договор кнопкой «Создать договор» с предзаполненными полями: объект, услуга, команда, даты этапов.

## Functional Requirements

### Входные параметры (форма над блоком)
- **Услуга** (`ДПИ` / `ЭАП` / `АЛР` / `Авторский надзор`) — обязательно.
- **Объект** — выбор из существующих или ввод параметров вручную:
  - Тип (`Жилой` / `Коммерческий`)
  - Сложность (`Стандартный` / `Средней сложности` / `Сложный`)
  - Площадь (м²)
  - Кол-во комнат (основные / вспомогательные / технические / ИИИ)
- **Длительность каждого этапа в раб. днях** — по умолчанию 20, редактируется (как при создании договора).
- **Желаемая дата старта** (опционально) — если указана, блок проверяет, реализуем ли договор с этой даты; если нет — сдвигается вправо до ближайшей возможной.

### Расчёт часов договора
- Используется существующая `calcStageHours()` из `lib/calc.ts` — для каждого этапа возвращает часы по ролям с учётом норм, базы, коэффициентов сложности.
- Коэффициент типа сотрудника `kT` пока не известен (команда ещё не назначена) — на этапе подбора используем `kT = 1.0` (Специалист) как baseline; после выбора конкретных людей пересчитываем.

### Подбор сотрудников и даты
Алгоритм (на старте — простой, без оптимизатора):

1. Для каждой роли из норм договора:
   - Отобрать всех сотрудников с этой ролью.
   - Для каждого сотрудника считать его свободные часы по месяцам (`hoursMonth − load[empId][monthKey]`) — данные из `calcLoad()` уже есть.
2. Для предполагаемой даты старта построить расписание этапов (`startDate + workingDays` цепочкой, как в существующих договорах).
3. Распределить часы каждого этапа по календарным месяцам (так же, как `calcLoad()`).
4. Для каждой роли подобрать сотрудника, у которого свободных часов в каждом из задействованных месяцев ≥ требуемых на эту роль на этом отрезке. Если не один — предложить варианты, отсортированные по:
   - меньшему `kT` (ведущий специалист → меньше часов нужно),
   - большему запасу свободной ёмкости.
5. Если для какой-то роли нет ни одного подходящего кандидата при выбранной дате — сдвинуть старт вправо на 1 неделю и повторить, до 6 месяцев вперёд.
6. Выбрать самую раннюю дату, при которой все роли закрыты.

### Результат блока
- **Карточка «Рекомендованный слот»** — крупно: дата старта, дата окончания, общая длительность.
- **Команда** — список ролей с подобранными сотрудниками; рядом значок свободной ёмкости каждого из них на интервале договора.
- **Альтернативы** — 2–3 запасных варианта (другие даты или другие составы), сворачиваются.
- **Полоса на таймлайне Block 1** — превью этапов нового договора пунктиром / полупрозрачно, чтобы было видно его положение относительно текущих этапов.
- **Кнопка «Создать договор»** — переход на `/contracts/new` с предзаполненной формой (`objectId`, `service`, `team`, `stages`).

### Если слот не найден за 6 мес
- Сообщение «В ближайшие 6 месяцев нет окна для этого договора».
- Подсказка: какая роль является узким местом (по которой не хватает часов).
- Кнопка «Показать тепловую карту по [роль]» — переход на `/heatmap` с фильтром.

## Data & Computation

- Все вычисления — на клиенте, без новых API.
- Используются существующие функции `lib/calc.ts`: `calcStageHours`, `calcLoad`, `addWorkingDays`, `countWorkingDays`, `getMonths`, `buildHolidaySet`.
- Нормы (`norms`) задают, какие роли участвуют в каком этапе и сколько часов это стоит.
- Учёт праздников — через `customHolidays` в `settings`.

## Possible Edge Cases

- **Нет норм для выбранной услуги** — кнопка «Подобрать слот» неактивна, подсказка «Заполните нормы для услуги X».
- **Нет сотрудников для какой-то роли из норм** — слот не подбирается, в результате видно «нет ни одного [роль]».
- **Все сотрудники уже перегружены** — показывается ближайший месяц, когда у узкого места появляется ёмкость.
- **Договор длится дольше 6 месяцев** — алгоритм всё равно подбирает старт; полоса на таймлайне обрезается по правому краю окна.
- **Объект введён вручную, не из БД** — кнопка «Создать договор» сначала создаёт объект, потом договор.
- **Желаемая дата старта в прошлом** — игнорируется, используется сегодня.
- **Один сотрудник подходит на несколько ролей одновременно** — нельзя его назначить дважды на один промежуток; алгоритм должен проверять конфликт.
- **Этап пересекается с праздником** — `addWorkingDays` уже учитывает `customHolidays`, дополнительной логики не нужно.

## Acceptance Criteria

- [ ] Старый блок «свободная ёмкость» полностью заменён новым «подбор слота» — ни таблица по ролям, ни вложенные сотрудники не отображаются.
- [ ] Форма параметров договора собирается и валидируется (услуга обязательна, либо объект из списка, либо ручные параметры).
- [ ] Кнопка «Подобрать слот» возвращает дату старта, дату окончания и команду по одному сотруднику на каждую роль.
- [ ] Подобранная дата ≥ сегодня и ≥ желаемой даты старта (если задана).
- [ ] Для каждого предложенного сотрудника на каждом месяце договора `свободные часы ≥ требуемые часы`.
- [ ] Если слот не найден — показывается узкое место (роль) и подсказка.
- [ ] На Block 1 видно превью этапов нового договора (визуально отличимо от существующих).
- [ ] Кнопка «Создать договор» переводит на форму создания с предзаполненными полями.
- [ ] Все вычисления используют существующие функции `lib/calc.ts`, нового API не появляется.

## Open Questions

1. **Несколько сотрудников на одну роль** — нужно ли поддерживать команду из 2+ людей на одну роль (например, два дизайнера делят 200 ч)? Или строго один человек на роль на договоре? нет, только один сотрудник на одну роль
2. **Преемственность с существующими договорами** — учитывать ли только `status = active` или и `done` тоже? (Сейчас `calcLoad` отбрасывает `done`.) закрытые договора не учитывать
3. **Альтернативы** — сколько вариантов показывать (2, 3, 5)? И по какому критерию ранжировать (раньше старт vs. больше свободного запаса vs. дешевле)? 2 варианта
4. **Выбор объекта** — нужна ли отдельная вкладка «новый объект» с вводом параметров, или достаточно списка существующих объектов? в большинстве случаев будет новый объект.
5. **Длительность этапов** — брать ли дефолт 20 раб. дней, или подставлять среднее по существующим договорам этой услуги? брать дефолт. вообще у нас всегда есть дефотный даты по каждому этапу каждого договора. Надо подумать, возможно стоит их вынести в раздел настройки. 
6. **Бюджетный фильтр** — нужен ли учёт стоимости команды (зарплата × часы) при ранжировании альтернатив? пока нет
7. **Превью на Block 1** — нужно ли реально подмешивать предложенные этапы в `calcLoad` (чтобы видеть, как они нагрузят календарь) или достаточно визуальной полосы? да, нужно видеть как они нагрузят календарь.
